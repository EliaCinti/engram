Everything you need to run wadachi: install it, wire it into your MCP client, understand how the brain thinks, and use all 31 tools well.

Install

Wadachi is a Python package (3.11+). The recommended install includes local semantic search (a ~30M-parameter embedding model that runs on your machine — no API, no cloud):

# with pipx
pipx install "wadachi[semantic]"

# or with uv
uv tool install "wadachi[semantic]"

# minimal (keyword search only, no ML dependencies)
pipx install wadachi

From source:

git clone https://github.com/EliaCinti/wadachi.git
cd wadachi && pip install -e ".[semantic]"

wadachi init

One command sets everything up:

wadachi init

It creates the brain directory (default ~/.wadachi; an existing legacy ~/.engram is detected and reused), brings the database to the latest schema (backing it up first if it already existed), writes SCHEMA.md (the wiki conventions — yours to edit), generates index.md, and registers the MCP server in Claude Code (via claude mcp add) and Antigravity automatically. It is idempotent: re-run it anytime.

flageffect
--brain-dir PATHput the brain somewhere else (e.g. an external SSD)
--no-claudeskip Claude Code registration
--no-antigravityskip Antigravity config

Verify: run wadachi doctor — it checks the brain dir, database integrity, schema version, files, search mode and MCP registration.

Any MCP client

Wadachi is a standard MCP server over stdio: command wadachi (no args), environment variable BRAIN_DIR pointing at the brain. That's the whole contract.

Multiple machines can share one brain (e.g. on an external drive or synced folder): point BRAIN_DIR at the same directory. One writer at a time is the safe pattern.

The brain on disk

Everything lives in one directory you own. Plain files, readable and editable by hand:

~/.wadachi/
├── SCHEMA.md      # the wiki conventions — edit it, it's yours
├── index.md       # generated catalog: one line per memory (don't edit)
├── log.md         # append-only history of operations (grep-able)
├── brain.db       # SQLite index: metadata, embeddings, beliefs
├── backups/       # automatic pre-migration + session backups
├── logs/          # server log (attach to bug reports)
├── global/        # cross-project memories (.md)
└── projects/<p>/  # per-project memories (.md)

The markdown files are the source of truth for content; SQLite is the authoritative index for metadata. Files carry OKF-conformant YAML frontmatter (type, title, project, tags, category, timestamps). The parser is tolerant: hand-edited or damaged frontmatter never breaks reading, and wadachi doctor --fix restores the canonical format from the database without ever touching your prose.

Memories & projects

A memory is one markdown file + one index row: title, project, tags, and a category — architecture, bugfix, config, pattern, context, reference, note.

Projects scope your knowledge. Register filesystem paths once (register_project) and get_context auto-detects which project you're in from the working directory. Searches scoped to a project also include global memories — and mark them scope: "global" so you always know whether evidence is project-specific or cross-cutting (evidence scoping).

Updates are non-destructive: every update_memory snapshots the previous version to memory_versions first. Nothing is ever silently lost — this powers both memory_history and time-travel (as_of).

Decisions

A decision records not just what you chose, but why, what you rejected, and in what context:

store_decision(
  decision     = "We use SQLite, not Postgres",
  rationale    = "local-first, zero-config, single file",
  alternatives = "Postgres rejected: operational overkill at this scale",
  context      = "storage layer choice for v1",
  project      = "myapp")

Decisions are typed nodes in the graph: cite one from a memory (“as per decision #7” or [[D7]]) and it becomes an edge. Later, why("why sqlite and not postgres?") returns the decision, its rationale, the rejected alternatives, and the memories that cite it as evidence.

Beliefs & time

A plain store treats every memory as true forever. A brain shouldn't. Every wadachi memory carries an epistemic envelope: confidence, status (active / stale / retired), an optional valid-until date, and superseded_by.

  • review_beliefs scans read-only for memories that have likely gone stale (superseded, past a temporal deadline, provisional wording) — they're also surfaced in every get_context under “needs review”.
  • flag_stale / set_belief let you confirm. Stale memories stay recoverable and are annotated in recall — never deleted.
  • Supersessions become typed graph edges: contradiction over time is structure, not noise.

Decay: memories never recalled slowly lose ranking priority (−2% per idle month, capped at 12%, transparently reported as a decay field). Any access rejuvenates them. Gentle by design — decay reorders, it never buries.

The typed graph

Wadachi builds a real knowledge graph over your brain — not a visualization, a data structure the system uses:

  • Nodes: memories and decisions (typed).
  • Citation edges from your writing: [[file-slug]] wikilinks, [[#42]] id links, “memory #42” / “decisione #42” in prose, [[D7]] for decisions — with typed relations (updates, contradicts, cites, relates, supersedes) inferred from surrounding words.
  • Semantic edges: nearest-neighbour similarity (automatic).
  • Entity edges: optional, via a Graphify entity graph (rebuild_entity_graph).

On top of it: recall_associative (spreading activation — strongly connected memories surface even when not textually similar), recall(neighbors=true) (1-hop typed neighbours on normal search), related_memories, memory_graph (stats + Mermaid), community detection for sleep, and a local visualizer: python -m wadachi.web.

Rule of thumb: link generously. Every [[link]] you (or your agent) write becomes an edge that associative recall, provenance and consolidation travel on.

LLM Wiki · Obsidian · OKF

The brain follows Karpathy's LLM Wiki pattern: an agent-maintained wiki (index.md catalog, log.md journal, SCHEMA.md conventions) made of interlinked markdown.

  • Obsidian: the brain dir is a valid vault — open it and you get the graph view for free. Run wadachi obsidian (explicitly, never automatic) to also translate id-style references into [[slug]] wikilinks Obsidian can resolve: it appends a marked Links section, never touches prose, versions every change, and supports --dry-run.
  • OKF: every file carries the Open Knowledge Format required type field — your brain is a conformant OKF bundle, portable to any OKF consumer. No lock-in, in any direction.

Start every session: get_context

The killer tool. One call loads project, relevant memories, recent decisions, what needs review, stats — and what the brain proposes — in a dense, pointer-based format (~400–600 tokens instead of thousands):

# wadachi · project: myapp · search: semantic
## memorie (rilevanza ↓)
#12 architecture ·0.81· CRDT merge strategy
D3  decision ·0.74· Yjs over Automerge
## da rivedere
#31 [temporal] past date next to a deadline
## il brain propone
💡 2 gruppi di memorie ridondanti da fondere → sleep()
stats: 104 mem · 21 dec · 6 prog
→ contenuto completo: expand_memory(ids=[…])
argmeaning
cwdworking directory → project auto-detection
task_descriptionwhat you're about to do → relevance ranking
max_tokens=600budget; truncates by relevance, never by age
format="json"full verbose payload if you need it

The brain proposes: pending insights awaiting your judgement, results of the last sleep run, or a nudge if sleep hasn't run in a while. The software proposes and explains — you decide. Nothing ever happens on its own.

Search & drill-down

recall(query) searches by meaning (or keywords without fastembed), returns compact results annotated with belief status, decay, and project scope. Add neighbors=true for each result's strongest typed graph neighbours. recall_associative(query) goes further: spreading activation over the graph, with the plain-cosine baseline returned alongside so the difference is auditable.

From any pointer, expand_memory(ids=[12, 31]) returns full contents (and counts as an access — it rejuvenates those memories against decay).

Store & link

store_memory(
  content  = "Redis pub/sub fan-out keeps sync under budget — see [[#12]]",
  title    = "Realtime fan-out choice",
  project  = "myapp",
  tags     = ["redis", "sync"],
  category = "architecture")
  • Store the moment you figure something out — 30 seconds now saves minutes next session.
  • Link related memories in the content ([[#id]] or [[slug]]): links are graph edges.
  • Record choices with store_decision — future why() depends on it.

Provenance & time-travel

why(question)

“Why do we use X and not Y?” → the matching decision(s) with rationale, rejected alternatives, context, date, and the memories that cite them in the graph.

as_of(date, query=None)

What did the brain believe on a given date? Memories that existed then — with content reconstructed as it was from the non-destructive version history — and which of them were already superseded or expired at that date.

as_of("2026-03-01", query="deploy pipeline")
# → memories alive on March 1st, content as of March 1st,
#   status_at_date: active / already superseded by #N / expired

Sleep & consolidation

Brains consolidate during sleep; so does wadachi — but it only proposes:

wadachi sleep        # CLI (cron-friendly) — or the sleep() tool
  • Merge candidates: graph communities (label propagation) whose memories are near-duplicates.
  • Decay candidates: leaves with no explicit links, never recalled, fading.
  • Orphans: disconnected nodes worth linking.

Nothing is modified. The report is cached, and your next get_context will propose it. To act: write your synthesis and call merge_memories(source_ids, title, content) — the new memory carries automatic [[#id]] provenance, sources are marked superseded (recoverable, always). consolidate() offers similarity-based candidates on demand, and reflect() proposes cross-project insights you can accept or reject.

Upgrading & safety

Your memories always survive an upgrade.

  • Schema changes ship as versioned migrations, applied on first start — after an automatic backup of brain.db to backups/. Failed migrations roll back.
  • File-format changes go through the tolerant parser + backfill (doctor --fix); prose is never rewritten.
  • Brains from any older version — including the Engram era (~/.engram) — are adopted in place. Nothing to export.
# the cautious path (recommended if your brain matters to you — it should):
wadachi export           # read-only portable snapshot, BEFORE anything migrates
pipx upgrade wadachi     # or: uv tool upgrade wadachi
wadachi doctor           # tells you if migrations are pending; they back up brain.db anyway

Coming from Engram? Install wadachi, run wadachi export immediately (it never touches the brain — no migrations run), put the archive somewhere safe, then proceed. Your ~/.engram brain is adopted in place; wadachi restore <archive> --replace can always bring you back — and it safety-exports the current state first, so even going back is reversible.

Troubleshooting

  • First stop: wadachi doctor. It checks everything and tells you what to do, including where backups are.
  • Logs: <brain>/logs/wadachi.log — every tool error with a full traceback. Set WADACHI_LOG=DEBUG for per-tool timings. Nothing is ever written to stdout (that's the MCP channel).
  • “Search feels dumb”: you're probably in keyword mode — install wadachi[semantic]. brain_status tells you the active mode.
  • recall_associative errors without fastembed: expected — it returns a clear message plus keyword fallback results.
  • Bug reports: open an issue with the doctor output + the last log lines — the template asks for exactly that.

FAQ

Does wadachi send my data anywhere?

No. No telemetry, no cloud calls, no exceptions. Embeddings are computed locally. The only network access in the whole product is pip downloading the package. This is a design rule, not a setting.

What runs when? Will it do things behind my back?

Nothing mutates on its own. The server responds to tool calls; migrations run at startup (with backup); index/log upkeep is bookkeeping on your own writes. Everything cognitive — merging, retiring, insights — is proposed and waits for you.

Can I edit memory files by hand?

Yes — they're yours. The tolerant parser handles imperfect frontmatter, and doctor --fix restores the canonical format without touching your text.

Where should the brain live?

Anywhere: BRAIN_DIR is the only pointer. External SSDs and synced folders work; keep one writer at a time.

How do I remove wadachi?

pipx uninstall wadachi, remove the MCP entry from your client, and your brain dir remains — plain markdown you can keep, grep, or open in Obsidian forever. Exit is as easy as entry. 轍
